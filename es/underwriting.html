REGA Arquitectura de seguro

Ya en 2001 nuestro equipo firmó un contrato para desarrollar un sistema de scoring y seguro para el primer banco hipotecario ruso DeltaCredit. Fue el momento en que nació iNSTANTLOAN y durante más de 15 años es el negocio principal de Bellwood Systems, la empresa matriz del proyecto REGA. Durante estos años los clientes han invertido en nuestro sistema más de 10 millones de dólares, incluyendo DeltaCredit, VTB, Svyznoy y Sberbank, el mayor banco minorista en Rusia (#56 Forbes) que actualmente utiliza iNSTANTLOAN para la venta de productos crowdsurance en más de 10.000 sucursales en todo el país. En 2014 introdujimos un servicio iNSTANTLOAN en la nube y ahora nuestros clientes pueden usar el sistema para las emisiones de préstamos de consumo y crowdsurance. Cada mes estamos procesando más de $ 25 millones en transacciones de varios clientes minoristas incluyendo MediaMarkt (grupo Metro). Siempre hemos soñado con desarrollar nuestro propio producto financiero y ahora se ha hecho posible gracias a las tecnologías blockchain (cadenas de bloques) y smart contracts (contratos inteligentes). Nuestra plataforma REGA es esencialmente el siguiente paso para el desarrollo del sistema iNSTANTLOAN que será rediseñado para incorporar la capa blockchain y estará disponible como software de código abierto y de servicio a la comunidad de desarrollo Fintech.

La plataforma REGA consta de dos partes principales: off-chain y blockchain. Off-chain es el sistema de scoring y suscripción rediseñado que incluye productos y bases de datos de clientes, el motor de coincidencia de productos y el módulo de scoring/gestión de riesgos. En blockchain tenemos varios tipos de contratos inteligentes: una estructura pool en cascada de tres niveles, incluyendo super pool, tokens y fábrica de smart contract. La plataforma utiliza el modelo de producto de tres niveles: (1) tipo de producto, (2) producto y (3) instancia de producto.

La segunda base de datos de la plataforma es el archivo de información del cliente. Cada cliente obtendrá un número de identificación único, su id, que vinculará todos los documentos electrónicos a la persona en particular. Almacenaremos todas las solicitudes presentadas por los clientes con los resultados de scoring, ofertas de productos y direcciones de smart contract. Toda la información sobre los casos presentados y los pagos se vincularán al cliente y utilizaremos esta información para los ajustes del modelo de scoring. La historia del cliente jugará un papel importante durante la evaluación de riesgos y nos facilitará una herramienta para calcular el precio individual para cada persona.

Para seleccionar el producto adecuado para el cliente necesitamos un mecanismo de coincidencia. No tiene sentido si usted tiene sólo un producto para vender, pero estamos creando una plataforma para muchos productos financieros y debemos proporcionar un servicio que ayudará al cliente a buscar la base de datos de productos. Como hemos descrito anteriormente, cada producto contiene reglas de concordancia que describen una condición para la cual podemos hacer una oferta. En estas reglas podemos usar atributos de datos demográficos del cliente, su ubicación, necesidades demandadas y los resultados de scoring. Si como resultado de la correspondencia, el cliente puede recibir varias ofertas, utilizaremos un nivel siguiente que coincida con la capa del tipo de producto, donde el desarrollador pueda especificar cómo presentar el resultado coincidente al cliente. Por ejemplo, si debemos mostrar solo una o varias ofertas, cómo clasificarlo o filtrar.

El módulo o servicio de gestión de riesgos será responsable del seguimiento del nivel de riesgo dentro de los grupos de riesgo. Usaremos el scoring para calcular el nivel de riesgo para cada cliente en particular y agruparemos clientes con riesgos equivalentes en sub-pools y pools. La gestión de riesgos utiliza modelos de scoring conductual para predecir el gasto de funcionamiento de un grupo de riesgo particular y comparar este gasto con el coste de liquidación del grupo para tomar una decisión sobre la próxima acción. Si el coste de liquidación del grupo es menor que el gasto de funcionamiento previsto, entonces el grupo de riesgo dado puede ser liquidado. En este caso los miembros del grupo que no presentaron casos recibirán sus contribuciones. Como resultado el modelo de riesgo se ajustará y las contribuciones para este rango de riesgo particular se elevarán para adecuar el nivel de riesgo ajustado. Si la cantidad de nuevas contribuciones calculadas es mayor que el atributo de contribuciones máximas de producto, entonces el nuevo afiliado no podrá recibir la oferta del producto y unirse a las pools.
